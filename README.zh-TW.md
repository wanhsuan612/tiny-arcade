## 為什麼做 Tiny Arcade

我很喜歡遊戲，也一直覺得「打開就能玩」是一件很迷人的事。

現在想試一款遊戲，常常要先經過商店、下載、帳號，甚至還沒開始玩就先遇到付費設計。我並不反對大型或商業遊戲，只是想替另一種體驗保留一個位置：**一個連結、一個想法，先玩再說。**

Tiny Arcade 就是我拿來實踐這件事的地方。這裡的遊戲刻意做得小，但不是隨便。我會反覆調整操作手感、美術、音效、手機體驗，以及那些看起來不起眼、實際上很影響「好不好玩」的細節。

我也不想把它做成只有我能維護的展示櫃。你可以來玩、挑毛病、提出腦中一直很想玩的遊戲，或直接一起做下一台機台。修一個 bug、補一段翻譯、調順一個動畫、加一個音效、提出一個怪點子，甚至完成一款新遊戲，都能讓這個遊戲櫃往前走。

## 我想守住的承諾

- **免費遊玩。** 我希望這裡帶來的是樂趣，而不是又一個結帳畫面。
- **點開就玩。** 不需要安裝、帳號、後端服務或建置流程。
- **規模小，不代表不用心。** 一款十分鐘的小遊戲，也可以有個性、手感與完整循環。
- **開放製作。** 想法、實驗、犯錯與修正都留在 repo 裡，讓大家看得見。
- **讓不同角色都能參與。** 玩家、美術、工程師、文字工作者、翻譯者，甚至第一次嘗試開源的人，都可以找到能幫忙的地方。

## 一起把遊戲櫃做大

### 來玩，然後誠實告訴我感覺

遇到 bug、操作不直覺、難度突然暴增、手機版跑掉，或只是某個細節「哪裡怪怪的」？請直接 [開一張 issue](https://github.com/ShemYu/tiny-arcade/issues/new)。截圖、錄影與真實反應都很有幫助。

### 許願一款你真的想玩的遊戲

腦中有一款「很想玩，但市面上找不到」的小遊戲嗎？可以直接 [建立一張 Game Wish 許願卡](https://github.com/ShemYu/tiny-arcade/issues/new?template=game-wish.yml)。

不用先寫完整企劃書。一句話的幻想、一個奇怪機制，甚至「某款遊戲，但我希望它可以……」都足以開始討論。許願是一個一起探索的起點，不是交付承諾；但好的點子，真的可能長成下一款遊戲。

### 直接參與製作

你不需要一次做完一款遊戲。以下都是很有價值的 contribution：

- 修 bug、改善無障礙或手機體驗；
- 打磨操作、動畫、UI、音效與效能；
- 補上繁體中文或英文內容；
- 改善遊戲素材與圖片處理工具；
- 提出新機制、關卡、敵人或平衡調整；
- 貢獻一款可以獨立執行的網頁遊戲。

可以先讀 [貢獻指南](./CONTRIBUTING.md#繁體中文)，較大的修改也很歡迎先在 issue 裡聊聊方向。

## 刻意保持簡單的架構

每款遊戲都有自己的路徑，直接以靜態檔案執行：

```text
games/<game-slug>/index.html
```

首頁的遊戲卡片、搜尋、分類與雙語內容，都由同一份 catalog 產生：

```text
assets/scripts/catalog.js
```

整個遊戲櫃沒有 framework、package manager、bundler、backend、登入系統或 runtime dependency。只靠 GitHub Pages 就能運作。

### 本機預覽

```bash
python3 -m http.server 8080
```

接著打開 `http://localhost:8080/`。

### 遊戲素材檢查

共用的 PNG 驗證工具放在 `tools/game-assets/`，每款遊戲則保留自己的風格規範、prompt、參考素材與 runtime manifest：

```bash
python3 tools/game-assets/validate_game_assets.py games/mochi-sky/asset-manifest.json
```

## 我會持續保護的原則

1. **先讓人玩到。** 訪客應該在一次點擊內進入遊戲。
2. **把小東西做完整。** 範圍可以很小，用心程度不必跟著縮水。
3. **技術要能被理解。** 第一次打開 repo 的人，也應該有機會快速看懂並開始修改。
4. **把回饋當成開發的一部分。** 玩家注意到的問題不是雜訊，而是設計資訊。
5. **保留意外與怪點子的空間。** 並不是每款遊戲都要追趨勢、套公式或綁上付費循環。

## 授權與素材標示

各遊戲與生成素材可能在自己的資料夾中附有不同的 attribution 或使用說明。若要將程式碼或美術移到本 repo 以外使用，請先閱讀對應遊戲的備註。
